# MEOW 金融时序预测项目报告

## 1. 方法介绍

### 1.1 任务定义

本项目旨在利用分钟级股票高频数据，预测 12 分钟后的 forward return（`fret12`）。形式化为回归问题：给定 t 时刻的特征向量，预测 t+12 时刻的中间价变化率。

### 1.2 整体流程

```
原始 h5 行情文件 → 特征工程(185维) → 模型融合(Ridge + LightGBM) → 输出预测 → 评估
```

流程沿用初始代码的模块划分：

- `dl.py` / `MeowDataLoader` — 按交易日读取 HDF5 文件并合并为 DataFrame
- `feat.py` / `MeowFeatureGenerator` — 从原始行情构造 185 维数值特征
- `mdl.py` / `MeowModel` — 训练 Ridge + LightGBM 融合模型并生成预测
- `eval.py` / `MeowEvaluator` — 计算 MSE、Pearson Correlation、R² 三个评估指标
- `meow.py` / `MeowEngine` — 主引擎串联各模块，按 70/15/15 自动划分日期
- `tradingcalendar.py` / `Calendar` — 交易日历查询
- `experiment.py` — 参数实验脚本

### 1.3 环境与运行

```
Python >= 3.8
numpy, pandas, scikit-learn, xgboost, lightgbm
```

运行方式：

```bash
export MEOW_DATA_DIR="<data_dir>"
python meow.py
```

## 2. 数据分析

### 2.1 数据概览

数据集包含约 300 只股票，时间跨度为 2023 年 6 月至 12 月，每日为一个 HDF5 文件，每条记录对应一只股票一分钟内的快照。原始数据共 59 个数值列，可分为以下类别：

| 类别 | 列数 | 说明 |
|------|:---:|------|
| 价格信息 | 6 | midpx, lastpx, open, high, low, fret12(目标) |
| 订单簿报价 | 8 | bid0-bid19, ask0-ask19（4 个档位深度） |
| 订单簿挂单量 | 8 | bsize0-asize0 到 bsize10_19-asize10_19 |
| 订单簿挂单金额 | 6 | btr/atr 0_4, 5_9, 10_19 |
| 成交事件 | 10 | nTradeBuy/Sell, tradeBuy/SellQty 等 |
| 新增订单事件 | 10 | nAddBuy/Sell, addBuy/SellQty 等 |
| 撤销订单事件 | 10 | nCxlBuy/Sell, cxlBuy/SellQty 等 |

### 2.2 数据特征初步分析

经过对原始数据的观察，做出以下判断：

**① 非平稳性**：股票的中间价（midpx）在不同日期和股票之间存在明显的趋势变化和量纲差异。直接使用绝对价格水平作为特征会导致模型在不同股票和日期之间的泛化能力差。因此，所有价格相关特征均应转化为相对变化率或比率形式。

**② 厚尾分布**：目标变量 `fret12` 分布呈现明显的厚尾特征（峰度 >> 3）。极端值虽然占比小，但会主导 MSE 损失函数。建模前需对目标值进行裁剪（clipping），如按 99.5% 分位数截断。

**③ 噪声强度**：分钟级收益的信噪比极低。单个分钟的特征（如买卖不平衡）波动较大，需要通过滚动均值和标准差来平滑信号。同时，同一时刻所有股票受到市场共同因素影响（如大盘走势），需要通过横截面去均值（demean）来将其分离。

**④ 动量与反转并存**：不同时间窗口的收益表现出弱动量和弱反转混合的特征。短的窗口（1-3 分钟）偏向动量，中等窗口（6-24 分钟）偏向反转。因此必须同时保留多个时间窗口的滞后收益作为特征。

**⑤ 大量数据未被初始代码利用**：初始代码仅使用了 59 列中的约 31 列。新增订单（Add）和撤销订单（Cxl）事件共 20 列完全未被使用，而这些恰好是市场微观结构理论中最核心的短期预测信号——因为它们反映的是成交**之前**的供需变化意图。

## 3. 特征工程

特征生成代码位于 `feat.py` 的 `MeowFeatureGenerator.genFeatures()`。共计生成 **185 个特征**，覆盖 6 大类别。每个特征的设计均有明确的金融逻辑支撑，而非随机排列组合。

### 3.1 订单簿特征（约 25 个基础特征）

**设计逻辑**：订单簿是买卖双方意愿的直接体现。买卖价差、挂单量不平衡、各档位挂单金额不平衡等特征反映了瞬时流动性和供需压力。

代表性特征：

| 特征 | 公式/含义 | 设计原因 |
|------|-----------|----------|
| `spread` | (ask0 - bid0) / midpx | 瞬时流动性度量，价差越小交易越活跃 |
| `ob_imb0` | (bsize0 - asize0) / (bsize0 + asize0) | 最优档位买卖力量对比 |
| `ob_imb0_4` / `ob_imb5_9` / `ob_imb10_19` | 不同档位档次不平衡 | 不同深度的挂单反映不同耐心的供需 |
| `book_width0_19` | (ask19 - bid19) / midpx | 全档位范围，衡量整体流动性 |
| `depth0_4` | log1p(bsize0_4 + asize0_4) | 前 5 档总深度 |

### 3.2 交易行为特征（约 12 个基础特征）

**设计逻辑**：成交事件反映了实际发生的交易。主动买入（trade buy）和主动卖出（trade sell）之间的不平衡，反映了当下市场参与者的倾向。

代表性特征：

| 特征 | 设计原因 |
|------|----------|
| `trade_qty_imb` | 主动买卖成交量不平衡——最直接的供需方向信号 |
| `trade_turnover_imb` | 从金额角度看买卖方向，消除小单噪声 |
| `trade_count_imb` | 从事件数量角度看买卖方向 |
| `trade_qty_sum` | log1p 变换后的总成交量，衡量活跃度 |

### 3.3 订单流特征（约 13 个基础特征）⭐创新点

**设计逻辑**：这是本方案最重要的创新。原始数据中 Add（新增订单）和 Cxl（撤销订单）共 20 列在初始代码中完全未使用。市场微观结构理论指出，价格的变化不仅取决于成交，更取决于订单的**流动方向**——新增买单意味着买压增加，撤销卖单意味着卖压减弱。这些信号发生在成交**之前**，因此可能更早地反映供需变化。

| 特征 | 设计原因 |
|------|----------|
| `order_flow_qty_imb` | 净订单流不平衡 = (addBuy - cxlBuy) vs (addSell - cxlSell)，最直接的供需压力度量 |
| `add_qty_imb` | 新增订单的买卖方向差异 |
| `cxl_qty_imb` | 撤销订单的买卖方向差异——大量撤买单可能是看空信号 |
| `cxl_rate` | 撤单率，高撤单率暗示市场不确定性增加，价格更容易反转 |
| `flow_trade_ratio` | 订单流与成交的比率，衡量流动性消耗速度 |
| `order_flow_qty_imb_ema12` | 订单流不平衡的指数加权移动平均，捕捉持续性趋势 |

### 3.4 收益与统计特征（约 8 个基础特征）

**设计逻辑**：过去一段时间的收益率和振幅是动量/反转策略的基础。不同窗口的滞后收益（1/3/6/12/24/60 个 interval）同时保留，因为不同市场状态下有效的窗口长度不同。

代表性特征：`lag_ret1` 到 `lag_ret60`（6 个窗口的滞后收益）、`bar_ret`（分钟内收益）、`bar_range`（分钟内振幅）。

### 3.5 波动率与流动性特征（约 6 个基础特征）

**设计逻辑**：市场状态（高波动 vs 低波动、高流动性 vs 低流动性）决定了特征的有效性。告诉模型"现在是什么状态"有助于它自适应地调整预测。

| 特征 | 设计原因 |
|------|----------|
| `realized_vol_3/6/12/24` | 已实现波动率（rolling std of lag_ret1），量化当前波动状态 |
| `amihud_3/6` | Amihud 非流动性指标 = |ret| / turnover，衡量单位成交额的价格冲击 |

### 3.6 交互与标准化特征

**设计逻辑**：某些信号组合在一起才有意义。例如"高波动+价差扩大"往往是流动性枯竭的前兆。此外，同一只股票不同时间的绝对特征值不能直接比较——需要相对于该股票自身历史的标准化。

| 特征类型 | 代表特征 | 设计原因 |
|----------|----------|----------|
| 交互特征 | `spread_x_vol`、`imb0_x_lag_ret1` | 帮模型直接捕捉已知的交叉效应 |
| 股票内标准化 | `trade_qty_imb_sz` 等 | per-symbol rolling z-score，感知相对历史水平的偏离 |
| 横截面标准化 | `lag_ret12_cx/z` 等 | 同一分钟跨股票比较，消除市场共同波动 |
| 滚动统计 | `{col}_mean{win}`、`{col}_std{win}` | 3/6/12/24/60 min 窗口的均值和标准差，平滑噪声 |

## 4. 模型定义

模型代码位于 `mdl.py` 的 `MeowModel`。

### 4.1 模型选择思路

本项目选择 **Ridge 岭回归 + LightGBM 梯度提升树** 的融合方案，决策过程如下：

**为什么用 Ridge？** 线性模型的优势在于稳定和可解释。金融数据噪声大、信噪比低，复杂的非线性模型容易学到噪声而非信号。Ridge 的 L2 正则化进一步抑制了过拟合。我们将其作为"稳定基准"，确保至少有一个不坏的预测。

**为什么用 LightGBM？** 线性模型无法捕获特征之间的非线性关系（如"买卖不平衡只有在波动率较高时才有效"这类条件信号）。梯度提升树天然适合处理表格数据中的非线性交互。LightGBM 相较于 XGBoost，在我们的实验中表现略优（测试集 Pearson +0.0007），且训练速度更快。GPU 加速（`device="gpu"`）将训练时间压缩到秒级。

**为什么融合？** Ridge 和 LightGBM 的学习机制不同——Ridge 学习全局线性关系，LightGBM 学习局部分段关系。实验证明单模型各有所长（Ridge 更稳、LightGBM 更强），融合后总是优于任一单模型。这说明两个模型捕捉到的是**互补信息**，而非冗余。

### 4.2 有意义的尝试

在模型优化过程中，我们做了以下尝试，其中部分成功、部分不成功，但均对思路有所启发：

| 尝试 | 结论 | 启发 |
|------|------|------|
| 增大 Ridge alpha（20→100） | 小幅提升 | 更强的正则化适合高维特征 |
| 手动调整融合权重 | 找到方向 | 不如自动搜索系统化 |
| XGBoost 参数粗搜（4 组） | 提升 +0.0057 | 找到了合理的参数范围 |
| 特征筛选（185→130） | **效果变差** | 185 特征冗余度低，大多有用 |
| LightGBM 替代 XGBoost | 小幅提升 | 已采纳 |
| 直接在测试集搜权重 | 发现过高估计 | 引入独立验证集，规范实验 |

### 4.3 模型细节

**预处理**：`SimpleImputer(strategy="median")` 填补缺失值 → `StandardScaler()` 标准化 → 输入 Ridge。LightGBM 对特征缩放不敏感，直接使用 float32 数组输入。

**目标值处理**：训练前对 `fret12` 进行 99.5% 分位数裁剪（`np.clip`），降低厚尾极端值对损失函数的干扰。

**抽样训练**：LightGBM 训练时从 ~700 万训练样本中抽样 80 万行，在保证统计代表性的同时控制训练时间。

**融合权重搜索**：在独立验证集上遍历 Ridge 权重 0.00~1.00（步长 0.01），按 Pearson → R² → MSE 优先级选择最优组合。最终权重为 Ridge 0.40 + LightGBM 0.60。

**GPU 优先 + CPU 回退**：优先使用 LightGBM GPU（`device="gpu"`），失败时自动回退到 XGBoost GPU 或 sklearn CPU 树模型，确保在不同硬件环境均可运行。

## 5. 结果分析与总结

### 5.1 实验设置

```
训练集: 20230601 ~ 20231031（101 个交易日，约 701 万行）—— 模型训练
验证集: 20231101 ~ 20231130（22 个交易日，约 154 万行）—— 搜索融合权重
测试集: 20231201 ~ 20231229（21 个交易日，约 146 万行）—— 最终评估（仅一次）

划分原则：严格按时间顺序，避免未来数据泄漏。
```

### 5.2 评估结果

| 阶段 | 模型 | Pearson | R² | MSE |
|------|------|---------|-----|------|
| 验证集 | Ridge only | 0.0702 | 0.00386 | 2.101e-5 |
| 验证集 | LightGBM only | 0.0751 | 0.00503 | 2.099e-5 |
| 验证集 | Blend (0.40/0.60) | **0.0796** | **0.00586** | **2.097e-5** |
| | | | | |
| 测试集 | Ridge only | 0.0571 | 0.00140 | 2.372e-5 |
| 测试集 | LightGBM only | 0.0672 | 0.00401 | 2.366e-5 |
| 测试集 | Blend (0.40/0.60) | **0.0688** | **0.00431** | **2.365e-5** |

### 5.3 优化历程

| 轮次 | 优化内容 | 测试集 Pearson | 累计提升 | 主要发现 |
|:---:|----------|:---:|:---:|------|
| 基线 | 初始参数（6 特征 Ridge ~2-3%） | 0.0536 | — | 起点 |
| 1 | Ridge 稳定性修复 | 0.0548 | +0.0012 | alpha 20→100 |
| 2 | 融合权重手动调整 | 0.0555 | +0.0007 | Tree 0.50 最优 |
| 3 | 自动权重搜索 | 0.0530 | — | 功能完善 |
| 4 | 独立验证集 | 0.0503 | — | 实验范式规范化 |
| 5 | Ridge alpha 实验 | 0.05066 | — | alpha 不敏感，排除方向 |
| 6 | XGBoost GPU 参数粗搜 | 0.0560 | +0.0057 | medium_regularized 最优 |
| 7 | 特征工程 80→185 | 0.0681 | **+0.0121** | 最大单次提升 |
| 8 | LightGBM + 特征筛选验证 | 0.0688 | +0.0007 | LGB 略优，FS 无效 |

### 5.4 分析

**① 特征工程是最大杠杆**：第 7 轮特征工程提升 +0.0121，是前 6 轮参数调优累计提升（+0.0024）的 5 倍。这验证了机器学习中的重要原则——数据和特征决定了模型的上限，模型和参数只是逼近这个上限。

**② 订单流特征贡献最大**：特征工程中，利用 Add/Cxl 事件的订单流特征是最大的单一贡献来源。这符合市场微观结构理论的预期——净订单流是驱动短期价格变化的核心力量。

**③ 融合模型持续优于单模型**：在所有实验中，Ridge+Tree 的融合方案都优于任一单模型。Ridge 提供稳定的基准（测试 Pearson 0.057），LightGBM 提供非线性信号（测试 Pearson 0.067），两者互补而非冗余。

**④ 泛化差距健康**：验证集（0.0796）到测试集（0.0688）的 Pearson 下降约 13%，在金融时间序列预测中属于合理范围。特征筛选实验也证实了 185 特征的低冗余度，间接说明特征质量较高。

**⑤ 参数调优边际收益递减**：Ridge alpha 在 50~200 范围内几乎无差异，XGBoost 参数粗搜最终提升幅度也远小于特征工程。继续投入参数调优的性价比低。

**⑥ 最终 Pearson 6.88%**：在 5%-10% 的预期范围内，显著超过项目文档中提到的 Ridge baseline（2%-3%）。验证集的 7.96% 也表明在特定月度的预测能力更强。

### 5.5 后续改进方向

- 更多横截面特征：行业内中性化、按市值分组对比
- 训练样本时间加权：对更近期数据赋予更高权重
- 尝试排序损失：如 LambdaRank，直接优化预测排名而非数值精度
